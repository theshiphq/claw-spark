# Configuration Reference

This document covers all the configuration files and options available in clawspark.

## skills.yaml

Location: `configs/skills.yaml`

This file controls which OpenClaw skills are installed and active. The format has two sections:

```yaml
skills:
  enabled:
    - name: skill-slug
      description: What this skill does

  custom:
    - name: my-skill
      source: https://github.com/user/repo
      description: What this skill does
```

**enabled:** Ships with clawspark defaults. Remove entries you don't want. Add entries from the OpenClaw skill directory.

**custom:** Your own skills or community skills not in the default list. The `source` field can be a GitHub URL or a local file path.

After editing, apply changes with:

```bash
clawspark skills sync
```

To scan installed skills for suspicious patterns (credential theft, obfuscation, exfiltration):

```bash
clawspark skills audit
```

To run a full system diagnostic when troubleshooting:

```bash
clawspark diagnose
```

## OpenClaw Configuration

Location: `~/.openclaw/openclaw.json`

This is the main OpenClaw configuration file. clawspark generates it during installation using `openclaw config set`. You can edit it with the same tool or directly.

Key fields generated by clawspark:

```json
{
  "agents": {
    "defaults": {
      "model": "ollama/qwen3.5:35b-a3b",
      "memorySearch": {
        "enabled": false
      }
    }
  },
  "gateway": {
    "port": 18789,
    "mode": "local",
    "auth": {
      "token": "auto-generated-during-install"
    }
  }
}
```

**Changing a value safely:**

```bash
openclaw config set agents.defaults.model "ollama/glm-4.7-flash"
clawspark restart
```

Use `openclaw config validate` to check your edits against the schema before restarting.

## Gateway Environment

Location: `~/.openclaw/gateway.env`

clawspark writes an env file that the gateway sources on startup. This tells OpenClaw how to reach Ollama:

```bash
OLLAMA_API_KEY=ollama
OLLAMA_BASE_URL=http://127.0.0.1:11434
```

## Environment Variables

clawspark respects the following environment variables. Set them in your shell profile or in `~/.openclaw/gateway.env`:

| Variable | Default | Description |
|---|---|---|
| `OLLAMA_BASE_URL` | `http://127.0.0.1:11434` | Ollama server URL |
| `OLLAMA_API_KEY` | `ollama` | Ollama auth key (any value works for local) |
| `OLLAMA_MODELS` | system default | Where Ollama stores model files |
| `WHISPER_MODEL` | `base` | Whisper model size (tiny/base/small/medium/large) |

## Changing Models After Install

You don't need to reinstall anything to switch models.

**Interactive selection:**

```bash
clawspark model switch
```

This shows you the recommended models for your hardware and lets you pick one.

**Direct switch:**

```bash
clawspark model switch qwen3.5:122b
```

If the model isn't downloaded yet, clawspark pulls it automatically (unless air-gap mode is enabled).

**Using any Ollama model:**

You're not limited to the models in `configs/models.yaml`. Any model from the Ollama library works:

```bash
clawspark model switch llama3:70b
```

clawspark will warn you if the model might not fit in your available memory, but it won't stop you.

**Check what's currently running:**

```bash
clawspark model list
```

This shows all downloaded models, the active model, and memory usage.

## Adding Cloud API Providers

By default, clawspark runs everything locally. If you want to add a cloud model as a fallback or for specific agents, you can add API keys via environment variables or the auth profile.

**Add a cloud provider via environment:**

Edit `~/.openclaw/gateway.env` and add your API key:

```bash
OLLAMA_API_KEY=ollama
OLLAMA_BASE_URL=http://127.0.0.1:11434
ANTHROPIC_API_KEY=sk-ant-...
```

Then create a separate agent that uses the cloud model:

```bash
openclaw agents add cloud-agent --model anthropic/claude-sonnet-4-20250514
```

**Important:** When cloud providers are configured, data sent to those agents will leave your machine and go to the cloud provider's servers. Your default local agent remains fully private. This is a deliberate tradeoff between privacy and capability. Only enable this if you understand and accept it.
